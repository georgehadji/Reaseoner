## 0. Executive Summary

**Reasoner** (Adaptive Reasoning Architecture) is a production-grade, event-sourced, multi-method AI reasoning orchestrator. It decomposes complex problems into structured multi-phase pipelines, leverages 90+ LLM models from diverse training ecosystems in parallel, applies independent critique, stress-tests solutions, and synthesizes actionable recommendations with epistemic labeling.

| Attribute | Value |
|-----------|-------|
| **Architecture Style** | Hexagonal DDD + CQRS + Event Sourcing + Mixin Composition [CONFIRMED] |
| **Backend** | Python 3.12+, FastAPI, Pydantic v2, ~120 source files |
| **Frontend** | Next.js 16 (App Router), React 19, TypeScript 5, Tailwind CSS v4 |
| **State Model** | `PipelineState` (~60 fields) + `PipelineAggregate` (event-sourced) |
| **LLM Routing** | OpenRouter primary (350+ models), 12 direct provider adapters, Ollama local |
| **Reasoning Methods** | 17 (multi-perspective, debate, jury, research, scientific, socratic, pre-mortem, bayesian, dialectical, analogical, delphi, CoVE, SoT, ToT, PoT, self-discover, writing) |
| **Presets** | 42 declarative presets (20 budget, 20 premium, 1 balanced, 1 experimental) |
| **Web Search** | SearXNG (self-hosted Docker) + Perplexity Sonar |
| **Memory** | Neuro L1/L2/L3 tiered cache with embedding search + Cohere rerank |
| **Security** | Input sanitization, prompt-injection defense, rate limiting, scoped auth, CSRF, circuit breaker |
| **Self-Healing** | AST-based introspection engine + auto test generation + CI/CD coverage gating |

## PHASE 2: ARCHITECTURE RECONSTRUCTION

### 2A. Structural View

#### 2A.1 Layered Architecture

The backend follows a **Hexagonal (Ports & Adapters) DDD architecture** with four explicit layers:

```
┌─────────────────────────────────────────────────────────────────┐
│                        API / INTERFACE LAYER                     │
│  FastAPI routes, SSE streaming, WebSocket, schemas, middleware   │
│  Files: api/__init__.py, api/routes/, api/streaming.py, etc.    │
├─────────────────────────────────────────────────────────────────┤
│                      APPLICATION LAYER                           │
│  EventBus, CQRS Handlers, Flows, Mixins (12), Services          │
│  Files: application/flows/, application/handlers/,              │
│         application/mixins/, application/services/               │
├─────────────────────────────────────────────────────────────────┐
│                        DOMAIN LAYER                              │
│  PresetCore, PresetRegistry — declarative routing configs       │
│  SaaS Entities — User, Subscription, Quota, Audit Log           │
│  Files: domain/preset_core.py, domain/preset_registry.py,       │
│         domain/saas.py                                           │
├─────────────────────────────────────────────────────────────────┤
│                         CORE LAYER                               │
│  Protocols, Constants, Settings, Events, Aggregates, Search,    │
│  Memory, Perspectives, Rerank                                    │
│  Files: core/protocol.py, core/constants.py, core/events/, etc.  │
├─────────────────────────────────────────────────────────────────┤
│                     INFRASTRUCTURE LAYER                         │
│  LLM Adapters, Persistence, WebSocket, Widgets, Translation,    │
│  Neuro, Healing                                                  │
│  Files: infrastructure/llm/, infrastructure/persistence/, etc.   │
└─────────────────────────────────────────────────────────────────┘
```

**Key insight:** The practical architecture is a **modular monolith** with streaming API interfaces. While the hexagonal layers are conceptually clean, the main execution path (`run_stream()` in `api/streaming.py`) instantiates `ARAPipeline` directly rather than routing through the CQRS `RunPipelineCommandHandler`. The CQRS layer is present but partially bypassed in the hot path.

```text
    Subagents ──► Application/Infrastructure
    Neuro ──────► Infrastructure + External embedding APIs
    Healing ────► Independent (reads all layers)
```

**Dependency Rule:** The hexagonal architecture enforces that dependencies point inward. The Domain layer has no dependencies on Application or Infrastructure. The Application layer depends on Domain and Core but not on Infrastructure (it uses ports). The Infrastructure layer implements the ports defined in Core.

**Reality Check:** There are several violations of this rule:
- `domain/preset_core.py` imports from `infrastructure.llm.registry` (`_REGISTRY`)
- `application/flows/__init__.py` imports from `api.serializers`
- `api/streaming.py` directly instantiates `ARAPipeline` rather than using CQRS handlers
- Mixins are tightly coupled to `PipelineState` internals

#### 2B.5 Streaming Event Flow (SSE)

```text
ARAPipeline.run()
    │
    ├──► Emit phase_start events → state.pending_events
    ├──► Emit phase_complete events → state.pending_events
    ├──► Subagents emit agent_start / agent_complete → state.pending_events
    └──► Yield pending_events as SSE data chunks

api/streaming.py
    │
    ├──► Receives generator from ARAPipeline.run()
    ├──► Formats events via serializers.py (_ser_0 through _ser_5)
    ├──► Yields SSE data: lines with "data: {json}\n\n"

ui-next/src/hooks/usePipelineStream.ts
    │
    ├──► Creates EventSource to /api/run
    ├──► Parses SSE data lines
    ├──► Dispatches events to Zustand store
    ├──► Updates phase timeline, chat messages, final synthesis
    └──► On [DONE]: saves conversation to IndexedDB
```

#### 3.5.2 Deferred Import Patterns (Cycle Avoidance)


**Analysis:** The codebase uses deferred (inline) imports to avoid circular dependencies. This is a code smell indicating tight coupling between layers. Specifically:
- `pipeline.py` defers importing `application.flows` because `flows/__init__.py` imports from `pipeline.py` (via mixin methods)
- `handlers.py` defers importing `pipeline.py` because `pipeline.py` imports from `application.mixins`

### 3.6 Hidden Coupling

#### 3.6.1 Layer Violations

| Violation | From | To | Impact |
|-----------|------|-----|--------|
| **Domain → Infrastructure** | `domain/preset_core.py` | `infrastructure/llm/registry.py` (`_REGISTRY`) | Preset validation depends on concrete registry. If registry format changes, domain breaks. |
| **Application → API** | `application/flows/__init__.py` | `api.serializers.py` | Flow registry imports serializers. Serializers should be API-layer only. |
| **Streaming → Pipeline internals** | `api/streaming.py` | `pipeline.py` (direct instantiation) | Streaming logic knows about `ARAPipeline` directly rather than using CQRS handlers. |
| **Snapshot → Aggregate internals** | `infrastructure/persistence/snapshots.py` | `aggregate._state_data` | Direct access to protected state breaks encapsulation. |

#### 3.6.2 Mixin Coupling

```text
ARAPipeline inherits from 11 mixins:
    SearchMixin, ArticlePipelineMixin, PerspectiveMixin,
```

```text
Two BaseLLMProvider classes coexist:
    1. infrastructure/llm/base.py — Legacy: (system_prompt, user_prompt) signature
    2. infrastructure/llm/ports.py — New: (messages: list[Message], config: LLMConfig) signature
```

### 3.7 Implicit Contracts

#### 3.7.1 PipelineState Field Naming Convention

```text
Implicit contract: Method-specific state fields MUST be named:
    "<method_name>_state"  (e.g., "jury_state", "debate_state")

The flow registry and mixins rely on this convention to store/retrieve method outputs.
Violating it causes silent data loss between phases.
```

#### 3.7.2 PhaseStep Naming Convention

```text
Implicit contract: ARAPipeline phase methods MUST be named:
    "_phase_<N>_<name>"  (e.g., "_phase_0_classify")

The flow registry binds method names to PhaseStep objects.
If a mixin defines a method with a different prefix, the flow won't discover it.
```

#### 3.7.3 Subagent Output Shape

```text
Implicit contract: All subagents MUST return PhaseSubAgentOutput with:
    - output: str (the generated content)
    - model: str (model ID used)
    - tokens: int (token count)
    - cost: float (cost in USD)
    - error: str | None (error message if failed)

The pipeline aggregates these into state.*_subagent_outputs lists.
Missing fields cause KeyError during serialization.
```

#### 3.7.4 Prompt Template Conventions

```text
Implicit contract: Phase prompt functions MUST return strings containing:
    - Language instruction (via get_language_instruction(state))
    - Problem wrapping (via _wrap_user_input())
    - JSON_ONLY_FOOTER for JSON-emitting phases

Deviation causes parsing failures in parsing.py.
```

#### 3.7.5 SSE Event Shape

```text
Implicit contract: SSE events MUST be JSON objects with:
    - type: str ("phase_start", "phase_complete", "chunk", "error", "done")
    - phase: str | None (phase key)
    - data: dict | str (event payload)

The frontend (usePipelineStream.ts) hardcodes these type strings.
Backend changes to event types require synchronous frontend updates.
```

#### 3.7.6 Synthesis Output Format

```text
Implicit contract: The synthesis phase MUST produce output with:
    - [SOLUTION]...prose with citations like [Title](url)...[/SOLUTION]
    - ```json...``` block with fields:
        * critical_insights: list[str]
        * action_blueprint: list[dict]
        * open_questions: list[str]
        * claim_labels: list[dict]
        * meta_audit: dict
        * sources: list[dict]

parsing.extract_solution_block() depends on this exact format.
```

## PHASE 4: DESIGN PATTERN & INTENT INFERENCE

### 4.1 Design Patterns

| Pattern | Status | Evidence | Intent |
|---------|--------|----------|--------|
| **Hexagonal Architecture (Ports & Adapters)** | [CONFIRMED] | `infrastructure/llm/ports.py` defines `LLMProvider` Protocol; `infrastructure/persistence/` implements event store port; domain has no infra imports (mostly) | Isolate domain logic from external concerns; enable provider swapping without domain changes |
| **Domain-Driven Design (DDD)** | [CONFIRMED] | Explicit `domain/` package with `preset_core.py`, `preset_registry.py`; `core/` contains domain events and aggregates; ubiquitous language in filenames | Model complex reasoning domain with explicit boundaries and entities |
| **CQRS (Command Query Responsibility Segregation)** | [CONFIRMED] | `application/handlers/handlers.py` has `RunPipelineCommandHandler`, `GetPipelineStatusQueryHandler`, `ResumePipelineCommandHandler` | Separate write and read paths; optimize each independently |
| **Event Sourcing** | [CONFIRMED] | `core/events/domain_events.py`, `core/aggregates/pipeline.py`, `infrastructure/persistence/event_store.py`; `PipelineAggregate` derives state from event sequence | Audit trail, temporal queries, reconstruct state at any point |
| **Mixin Composition** | [CONFIRMED] | `ARAPipeline` inherits from 11 mixins in `application/mixins/` | Compose method-specific behaviors without deep inheritance hierarchies |
| **Provider Router with Fallbacks** | [CONFIRMED] | `infrastructure/llm/router.py` — explicit fallback tables per role; `ProviderRouter.call()` with fallback chain | Resilience through cross-lab diversity; automatic recovery on provider failure |
| **Circuit Breaker** | [CONFIRMED] | `circuit_breaker.py`; used in `ProviderRouter` and `core/rerank.py` | Prevent cascade failures; fail fast when external service is unhealthy |
| **Token Bucket Rate Limiter** | [CONFIRMED] | `rate_limiter.py`; applied in `api/__init__.py` middleware | Protect against abuse; enforce per-client quotas |
| **Strategy Pattern** | [CONFIRMED] | `infrastructure/persistence/snapshots.py` — `SnapshotStrategy` with version/time/phase/size-based implementations | Pluggable snapshot strategies without changing aggregate logic |
| **Factory Pattern** | [CONFIRMED] | `infrastructure/llm/registry.py` — `build_provider()` creates provider instances from model ID | Decouple provider creation from usage |
| **Singleton (EventBus)** | [CONFIRMED] | `application/event_bus/bus.py` — `get_event_bus()` returns global singleton | Single event bus instance across the application |
| **Observer Pattern** | [CONFIRMED] | EventBus with typed subscribers; `agent_start`/`agent_complete` events | Decouple event producers from consumers |
| **Template Method** | [LIKELY] | `hypergate/base_sub_agent.py` — `BaseSubAgent` defines `run()` skeleton, subclasses override `_build_prompt()` and `_parse_response()` | Standardize sub-agent lifecycle while allowing customization |
| **Chain of Responsibility** | [LIKELY] | `ProviderRouter` fallback chain — primary → explicit fallback → default | Try multiple providers in sequence until one succeeds |
| **Proxy Pattern** | [LIKELY] | `api/__init__.py` mounts `neuro/server.py` router; API routes proxy to backend services | Add cross-cutting concerns (auth, rate limiting) without modifying services |
| **Repository Pattern** | [LIKELY] | `infrastructure/persistence/event_store.py` — abstracted event storage; `postgres_store.py` is alternate implementation | Abstract persistence details behind repository interface |
| **Unit of Work** | [SPECULATIVE] | EventStore appends events in batches; SnapshotManager coordinates snapshot writes | Ensure atomicity of state changes |
| **MVC / MVVM** | [SPECULATIVE] | Frontend separates components (View), stores (ViewModel/Model), hooks (Controller) | UI separation of concerns; though not strictly MVC |
| **Plugin Architecture** | [LIKELY] | `infrastructure/widgets/discover.py` — widget registry; phases are pluggable modules | Extend functionality without core changes |
| **Lazy Loading** | [CONFIRMED] | `postgres_store.py` lazy-loads asyncpg; `healing/` only runs in CI | Defer expensive initialization until needed |

### 4.2 Architectural Style

| Style | Status | Evidence | Trade-offs |
|-------|--------|----------|------------|
| **Modular Monolith** | [CONFIRMED] | Single Python package (`src/reasoner/`) with internal module boundaries; single FastAPI process; single Next.js frontend | Simpler deployment than microservices; modules communicate in-process; risk of tight coupling if boundaries erode |
| **Event-Driven (partial)** | [CONFIRMED] | EventBus, DomainEvents, SSE streaming; but main path bypasses event bus for direct pipeline calls | Good for audit trails and side effects; adds complexity; current implementation is partially bypassed |
| **Streaming-First API** | [CONFIRMED] | SSE endpoints for `/api/run`, `/api/run-followup`; WebSocket for real-time progress | Low latency UX; harder to cache and retry; requires careful connection management |
| **Pipeline-Orchestration** | [CONFIRMED] | `ARAPipeline` with explicit phase sequences; method-specific flows via `build_default_flow_registry` | Clear reasoning steps; extensible for new methods; central orchestrator becomes a bottleneck |
| **Self-Healing System** | [CONFIRMED] | `healing/` with introspection engine, test generation, CI/CD feedback loops | Proactive quality maintenance; risk of false positives and noise |

### 4.3 Trade-offs and Constraints

| Trade-off | Decision | Rationale | Cost |
|-----------|----------|-----------|------|
| **Event Sourcing vs. Simplicity** | Event sourcing implemented but main path bypasses it | CQRS handlers exist for future extraction; current hot path optimizes for latency | Audit trail is incomplete for direct pipeline runs; snapshot logic adds I/O overhead |
| **Mixin vs. Service Composition** | Mixins chosen over injected services | Mixins provide easy access to shared state; reduces boilerplate | ARAPipeline becomes a god object (902 lines + 11 mixins); test isolation is harder |
| **SQLite vs. PostgreSQL** | SQLite default, PostgreSQL optional | Zero-config local development; SQLite is sufficient for single-node deployments | SQLite `ThreadPoolExecutor` may bottleneck under high concurrency; no replication |
| **OpenRouter vs. Direct APIs** | OpenRouter primary, direct adapters secondary | Single API key for 350+ models; simplified routing | Higher latency (proxy hop); vendor lock-in risk; cost markup |
| **SSE vs. WebSocket** | SSE for pipeline, WebSocket for progress | SSE works over HTTP (firewall-friendly); automatic reconnection | Bidirectional communication requires separate endpoint; cannot push from server without client request |
| **Temperature Whitelist** | Explicit whitelist for temperature-supported models | Some models (OpenAI, Anthropic) reject temperature parameter | Must update whitelist for every new model; silent failures if missed |
| **Deferred Imports** | Inline imports to avoid circular dependencies | Faster to implement than restructuring dependencies | Hides coupling; harder to trace dependencies; potential runtime surprises |
| **Windows Platform Patch** | Monkey-patch `platform` module for OpenAI SDK | Prevents WMI deadlock on Windows | Fragile; breaks with OpenAI SDK updates; global side effect |
| **JSONL for Memory** | TaggedMemory uses JSONL files | Append-only, human-readable, no schema migration | No querying beyond linear scan; concurrent writes need locking; file size grows unbounded |
| **Preset Hardcoding** | 42 presets defined in Python dict | Fast lookup; type-safe; version controlled | Requires code change to add/modify presets; no runtime customization |

## PHASE 5: EDGE CASE + FAILURE ANALYSIS

### 5.1 Bottlenecks

| Bottleneck | Location | Trigger | Impact | Mitigation |
|------------|----------|---------|--------|------------|
| **ARAPipeline single-threaded execution** | `pipeline.py` — `run()` is synchronous generator | Any pipeline request | One pipeline per process thread; concurrent requests block each other | Run multiple uvicorn workers; extract to async task queue |
